01 — The real problem

Finding genuine leads is harder than it looks

Every platform we wanted to use had its own set of restrictions. Before getting to leads at all, we first had to figure out how to even collect data reliably from each one.

Platform restrictions we ran into
💬 Reddit

Reddit has a strict API policy that changed in 2023. They started charging for API access and put hard rate limits on how many requests any app can make per minute. The free tier only allows 100 requests per minute and beyond a certain volume, Reddit starts returning empty results or throwing errors entirely. On top of that, Reddit's search algorithm deprioritises older posts heavily, so searching by keyword mostly surfaces recent popular posts rather than actual discussions where real people describe their problems. Scraping without the official API gets IP-banned quickly.

𝕏 Twitter / X

After Elon Musk's acquisition in 2022, Twitter's free API was shut down completely. The basic tier now costs $100 per month and only allows reading 10,000 tweets per month, which is nowhere near enough for daily lead discovery. The free endpoints that still exist are severely rate-limited and return incomplete data. Scraping tweets without the API gets blocked via JavaScript rendering challenges that require a real browser session, and even then, IP bans come in fast if requests look automated.

💼 LinkedIn (hardest)

LinkedIn has by far the most aggressive anti-scraping system. Their official API gives almost no useful data unless you have a partner-level agreement. Any non-residential IP (cloud servers, VMs, datacenters) gets instantly blocked. They use browser fingerprinting, CAPTCHA, and behaviour analysis to detect automation. Session cookies expire and get invalidated the moment something looks non-human. Even running Playwright (a real browser) on a cloud server gets caught because the IP itself is flagged. The only way to get reliable data from LinkedIn is a real browser session running from a home or office internet connection.

What we built first and why it did not work
The n8n approach

The first version used n8n, a no-code automation tool, connected to a set of third-party scrapers and data APIs. We used Hunter.io to find email addresses, ScrapingDog to pull page content from sites that blocked direct requests, and SerpAPI to run Google search queries and get structured results back. The idea was to search across platforms for people who needed software development help.

The search queries looked like this: "need a software developer", "looking for app developer", "hire automation developer", "need someone to build". Run these across LinkedIn, Reddit, Twitter, and various forums and you would expect to find real clients.

What actually came back was mostly agencies. Development shops, freelancer platforms, outsourcing companies and hiring firms have flooded the internet with exactly these keywords because those are the terms their clients search for. So every search we ran returned pages of agencies advertising their own services, developers promoting their portfolios, and job listings. We ran multiple pipeline runs and could not find a single genuine lead through this approach.

The problem was fundamental. Real clients do not write "I need a software developer". They write "our inventory is managed in three separate spreadsheets and we have 40 staff now and it's becoming unmanageable" or "we're processing 200 client requests a month manually and losing track of half of them". They describe a business situation. No keyword search would ever surface those posts.

The approach that actually worked
Keyword search (what we did first)
Searching for "hire developer" type phrases
Every major platform's search index gets gamed by agencies. The people who write "hire developer" in their posts are the ones selling development services, not buying them.
  • Results almost entirely agencies and freelancer platforms
  • Developers marketing themselves showed up constantly
  • Job listings filled every results page
  • Real buyers never phrase their problem as a search keyword
  • Zero genuine client leads found across multiple runs
Feed browsing (OpenOutreach method)
Scrolling through feeds like a human would
Discovered OpenOutreach, an open-source project that takes a completely different approach. Instead of searching, it browses feeds the same way a person would, collecting posts as they appear.
  • Feed posts come from actual founders and business owners
  • People describe real problems they are currently dealing with
  • Full context is visible including who they are and what they run
  • No keyword gaming possible since there is no keyword being searched
  • Lead quality improved dramatically on the very first test run
Where the AI comes in, and why it is not 100% reliable

Feed browsing collects a lot more relevant posts than keyword search, but it still collects everything. Developers talking about their own work, founders who happen to be technical, success stories, industry news. All of it ends up in the collected data. That is where the LLM scoring step matters. Every collected post gets fed to the AI model, which reads it and scores it from 1 to 10 based on whether this looks like a genuine software development opportunity.

Is it 100% reliable? No. The AI makes mistakes, particularly on edge cases. A technical founder who writes about scaling infrastructure challenges might get scored as a developer when they are actually a potential client. A post describing a business problem without explicitly mentioning software might get scored lower than it deserves. The model is good on clear-cut cases but ambiguous posts still slip through or get incorrectly rejected.

The way to fix this over time is training. Every time someone accepts or rejects a lead in the dashboard review panel, that decision gets saved. The system uses those decisions to build a model that understands what kinds of posts are actually relevant for Demaze specifically, not just in general. The more decisions get recorded, the smarter the pre-filtering becomes, and the fewer mistakes the pipeline makes on the ambiguous cases. It improves continuously the more it gets used.

02 — The journey

From n8n to a production Python system

Every version of this system taught us something the previous one could not. Here is the full progression from start to where it is today.

v1 · n8nStarted with n8n, which worked until it did not

n8n was a sensible starting point. It lets you build automation visually without writing backend code, and it connected to LinkedIn search, basic scoring via webhooks, and Supabase for storage. Good enough to get something running fast. The problem appeared as soon as we needed anything more sophisticated. Custom scoring logic, ML models, concurrency control, feedback loops from human review decisions. n8n is not built for any of that. Every edge case required a workaround that created three new problems.

What changedRebuilt from scratch in Python using FastAPI. Full control over every part of the system, proper async execution, and the ability to plug in ML components like the GP scorer without fighting against the framework.
v1 and v2Keyword search returned agencies instead of clients, every single time

Both in n8n and in the early Python version, we searched LinkedIn with phrases like "hire developer" or "need software built". The results were dominated by agencies self-promoting, developers posting about their own work, and job listings. We ran multiple complete pipeline runs. Not a single genuine client lead came through. After digging into why, the pattern became clear. Real buyers describe a business situation. They never phrase it as a search keyword because they are not trying to be found.

What changedFound OpenOutreach, an open-source project that scrolls the LinkedIn feed the way a person would browse it instead of searching with keywords. Adapted that approach using Playwright browser automation. The very first test run surfaced noticeably better posts.
v2 · scoring speedScoring 60 posts was taking over 5 minutes and kept stalling

The original scorer sent one post at a time to the Groq API, with a 2.1 second forced sleep after every call to stay under the rate limit. 60 posts meant 60 calls and 126 seconds of pure waiting before any results came back. Even after adding concurrency, hitting Groq's 30 requests per minute limit caused retry loops that stalled the pipeline for minutes. Switching to NVIDIA helped but 6 concurrent calls still exceeded their free tier limit.

What changedBatch scoring. Five posts go into one API call and five scores come back in a single JSON array. 60 posts became 12 API calls instead of 60. Combined with asyncio.gather() running multiple batches concurrently and a Semaphore(3) to stay within rate limits, total scoring time dropped to around 45 seconds. NVIDIA Nemotron-49B became the primary model with Groq as automatic fallback when it rate-limits.
v2 · visibilityLinkedIn ran silently for 5 minutes with nothing showing on screen

The LinkedIn scraper runs as a separate browser subprocess because Playwright needs to control an actual browser. The original code used subprocess.run() which holds all output until the process finishes. During the 3 to 5 minute LinkedIn browser run, the dashboard showed nothing at all. Anyone watching assumed the pipeline had crashed or frozen. There was no way to tell it was actually working.

What changedReplaced subprocess.run() with subprocess.Popen() and switched to reading output line by line as it arrives. Each line gets forwarded immediately to the async event loop using loop.call_soon_threadsafe(). Every scroll action, every post collected, now appears in the dashboard log in real time as it happens.
current limitationLinkedIn requires a personal machine and cannot run on cloud servers

When the system was deployed to Railway, LinkedIn stopped working entirely. LinkedIn's infrastructure blocks all cloud datacenter IP ranges at the network level. AWS, Railway, Render, Google Cloud, all of them are in LinkedIn's block list. It is not a policy issue that can be worked around. Even if you get past that, the session cookies used for authentication get invalidated the moment any request comes from a datacenter IP.

Current setupLinkedIn runs on a local PC using a home internet connection, which uses a residential IP that LinkedIn treats as normal. Cloudflare Tunnel runs alongside it and exposes the server with a permanent public HTTPS URL so the dashboard works from anywhere. Reddit and Twitter run on Railway without any issues since they do not have this restriction.
ongoingAI scoring is good but not perfect, especially on ambiguous posts

Feed browsing is a significant improvement over keyword search but the AI model still makes mistakes. A technical founder writing about scaling challenges can look like a developer to the model. Posts that describe a clear business problem without explicitly mentioning software get scored inconsistently. These edge cases require human review and the model cannot get better on its own without that input.

How it improves over timeEvery accept or reject decision made through the dashboard review panel gets saved. The GP scorer uses these decisions to learn what kinds of posts are actually relevant for this specific use case. Once 10 decisions have been recorded it activates and starts pre-filtering posts before they reach the LLM at all. The more it gets used, the fewer mistakes it makes on exactly the posts where the LLM currently struggles.

What actually happens when you click Run
step 1
Dashboard talks to the backend via FastAPI + SSE

The dashboard is a plain HTML file running in your browser. When you click Run, it sends a POST request to /api/pipeline/run on the FastAPI server. FastAPI is a Python web framework that runs on top of uvicorn, which is an async server. Instead of processing the pipeline inline and making you wait, FastAPI registers it as a background task and immediately replies to the browser so the page stays responsive. Separately, the browser opens an SSE connection to /api/stream. SSE stands for Server-Sent Events. It is a one-way channel where the server can push text lines to the browser over a single HTTP connection that stays open. Every log line you see in the dashboard arrives this way in real time, without polling.

step 2
Reddit and Twitter scraped in parallel using asyncio + SerpAPI

Python's asyncio library lets you run multiple tasks at the same time without using threads. asyncio.gather() takes both scraper functions and starts them simultaneously. While one is waiting for SerpAPI to respond, the other is doing the same. SerpAPI is a third-party service that handles Reddit and Twitter's rate limits and anti-bot restrictions on our behalf. We send it a search query, it returns structured JSON with post text, author, and URL. We never talk to Reddit or Twitter directly, which avoids the API restriction problems mentioned earlier. Both scrapers finish in under 30 seconds and return around 40 to 60 posts combined.

step 3
LinkedIn controlled by Playwright running a real Chrome browser

Playwright is a browser automation library from Microsoft. It launches a real Chrome window, loads LinkedIn using your saved session cookies, and scrolls through the feed exactly the way a person would. It pauses between scrolls, waits for posts to load, and extracts the text from each one. LinkedIn cannot easily distinguish this from a real human because it is a real browser making real requests from a residential IP. The scraper runs as a separate subprocess because Playwright blocks while it runs. To stream its logs to the dashboard in real time, we use subprocess.Popen() instead of subprocess.run(), which lets us read output line by line as it arrives. Each line is passed to the async event loop using loop.call_soon_threadsafe() so it appears on your dashboard immediately without waiting for the full scrape to finish.

step 4
Deduplication by post ID, then regex hard reject

Every post from every source has a unique ID. The system stores every ID it has ever seen in Supabase. Before scoring, it checks each incoming post ID against that stored list and drops any duplicate. This stops the same post from being scored again on a second run. Then the hard reject filter runs. It uses regex patterns, which are text matching rules, to catch obvious junk without any AI involved. Rules like "we are hiring", known agency account name patterns, and news article structures. Regex runs instantly and costs nothing. Removing 30 to 40 percent of posts here directly cuts the number of paid API calls in the next steps.

step 5
GP scorer: FastEmbed turns text into numbers, Gaussian Process makes the prediction

This step only runs after you have made 10 accept or reject decisions in the dashboard. Here is the full chain of what happens. FastEmbed is a lightweight library that converts a piece of text into a list of 384 numbers, called an embedding. These numbers capture the meaning and context of the text mathematically. Similar texts produce similar number patterns, different ones produce very different patterns. Once each post is converted into numbers, a Gaussian Process model compares those numbers against the posts you have already labelled. A Gaussian Process is a machine learning model that estimates similarity, not just pattern matching. It says "this new post is mathematically close to posts you rejected before, so the probability you want this is low." Posts below 0.2 probability are dropped before reaching the AI. The more you label, the more accurate the comparison becomes.

step 6
LLM batch scoring via NVIDIA Nemotron with Groq as automatic fallback

The remaining posts are split into groups of 5. Each group is sent to the NVIDIA NIM API with a prompt that tells the model to score each post from 1 to 10 based on how clearly the person needs software built, and to return a JSON array with the score and a reason for each one. Scoring 5 posts in one call instead of one at a time cuts API calls from 60 down to 12. All 12 batches are submitted at the same time using asyncio.gather(), but a Semaphore limits how many actually run at once to 3, which keeps us within free tier rate limits. A Semaphore is basically a counter that blocks new requests from starting until one of the current ones finishes. If NVIDIA returns a 429 rate limit error, the same batch is automatically retried against the Groq API using Llama-3.3-70B. The caller never needs to know which model handled it.

step 7
Qualified leads saved to Supabase in parallel, emails looked up via Hunter.io

Posts scoring 7 or above are saved to Supabase, which is a hosted PostgreSQL database with a REST API. All save calls fire simultaneously using asyncio.gather() so writing 6 leads takes the same wall time as writing 1. At the same time, Hunter.io receives the company domain for each lead and tries to find a publicly listed email address for someone at that company. Hunter.io works by crawling the web for email addresses associated with domains and maintaining a database of what it finds. When the run completes, the dashboard polls for new leads and displays them in the review panel. Every accept or reject you click hits the /api/feedback endpoint, which adds that post and your decision to the GP scorer's training data, making step 5 smarter on the next run.

Stage 1 and 2: Scrape and deduplicate

Reddit and Twitter scrape in parallel using SerpAPI, which handles their rate limits and returns structured results. LinkedIn scrolls through the feed using a real Playwright browser session. All three run and return posts, which then get merged and deduplicated using the post ID so the same post never gets scored twice across runs.

Stage 3: Hard reject

Before any post reaches the AI model, a set of regex patterns filters out the obvious junk. Posts from known agency account patterns, posts containing "we are hiring", posts that are clearly news articles or tutorials. This stage typically removes 30 to 40 percent of collected posts without spending a single API call, which directly reduces scoring cost and time.

Stage 4: GP pre-rank (once active)

Every time you click Accept or Reject on a lead, the system remembers that decision. After 10 clicks, it starts learning your pattern — what kind of posts you tend to keep versus throw away. It converts each post into a list of numbers that captures the meaning of the text, then uses those numbers to build a rough sense of your taste. When a new post comes in, it checks how similar that post is to things you already accepted or rejected and makes a guess about whether you would want it. If it is pretty confident you would reject it, it skips sending that post to the AI entirely. This saves time and cost on content that is obviously not useful. The more you click, the better the guesses get.

Stage 5, 6 and 7: Score, qualify and save

Remaining posts go to the AI model in batches of 5. The model scores each one from 1 to 10. Posts scoring 7 or above get saved to Supabase as qualified leads. The save calls all run concurrently using asyncio.gather so saving 6 leads takes the same time as saving 1. Hunter.io runs in parallel to look up contact emails for leads where a company domain can be identified.

Batch scoring — the key optimisation
Why 5 leads per call design decision
Sending each lead individually meant 60+ API calls per run, constant rate limit hits, and sequential waits. Batching 5 leads into one prompt with a structured JSON response format cuts calls to 12 and fits comfortably within any free-tier limit. The model returns a JSON array — one score and reasoning per lead — which gets parsed and matched back to the original posts.
60 leads → 12 calls → fits 30 RPM free tier with room to spare
Concurrency model
asyncio.Semaphore(3) limits simultaneous API calls. asyncio.gather() runs all 12 batches concurrently within that limit. A single shared httpx.AsyncClient is reused across all calls, eliminating per-call TCP connection overhead. The same gather pattern applies to Supabase writes — all leads are saved in parallel, not sequentially.
Semaphore(3) × 12 batches = wall time of ~4 batches sequentially
SSE vs WebSocket design decision
Log streaming uses Server-Sent Events over a persistent HTTP connection. SSE is unidirectional (server → browser only), which is exactly what's needed here. No handshake overhead, no socket management, works through proxies, and the browser's EventSource API handles reconnection automatically. WebSockets would add complexity for no benefit.
Single HTTP connection, EventSource handles reconnect
Self-learning GP scorer
Every time you click Accept or Reject on a lead, the system remembers that decision. After 10 clicks it starts learning your pattern. It checks how similar each new post is to things you already accepted or rejected and makes a guess about whether you would want it. If it is confident you would reject it, that post never reaches the AI at all. The more you use it, the better the guesses get.

# 5 leads per call. Model returns a JSON array.
def _build_batch_prompt(leads: list) -> str:
    SYSTEM = """Score each post 1-10 for software development intent.
Immediately score 1 if the author is a developer, agency, writing a success
story, posting a job, or producing promotional content.
Score 9-10: explicitly requests software to be built.
Score 7-8:  business problem that clearly requires a software solution.
Return JSON array only: [{"id":0,"score":8,"reason":"..."},...]"""

    posts_block = "\n\n".join(
        f"POST {i}: {lead['text'][:800]}"
        for i, lead in enumerate(leads)
    )
    return f"{SYSTEM}\n\n{posts_block}"

# Primary: NVIDIA. Fallback: Groq. Transparent to caller.
async def _score_batch(leads: list) -> list:
    async with _sem:  # Semaphore(3)
        result = await _call_nvidia(leads)
        if result is None:   # 429 or error
            result = await _call_groq(leads)
        return result

# All batches run concurrently within the semaphore limit
batches = [leads[i:i+5] for i in range(0, len(leads), 5)]
results = await asyncio.gather(*[_score_batch(b) for b in batches])

04 — AI models

Current setup and what a better model unlocks

The system is model-agnostic by design. Swapping providers is a one-line change. Here is where the current model does well and where it falls short.

Currently in production
NVIDIA Nemotron-49B
nvidia/llama-3.3-nemotron-super-49b-v1.5 · free via NVIDIA NIM
  • No billing required — starts immediately on a free API key
  • Handles the 5-lead batch format reliably, consistent JSON output
  • Good baseline accuracy on clear-cut cases (explicit "build me X" posts)
  • Falls back to Groq Llama-3.3-70B automatically on 429
  • Struggles with indirect signals and ambiguous founder context
  • Context window caps how long each post can be before trimming
Where Claude / Gemini would improve this
Claude Sonnet or Gemini Pro
Anthropic / Google · paid or generous free tiers
  • 200k token context — can pass the full post, author bio, and company info without trimming
  • Much better at indirect signals: "we're manually running this in spreadsheets for 200 clients" → lead, not just noise
  • Claude follows structured scoring instructions with notably fewer format errors
  • Gemini Pro: 1 million tokens/day free — easily enough for current volume without billing
  • Both support system-level calibration prompts that can be tuned from real false-positive data
  • Better false-reject rate on technical founders who also happen to write about code
Why not switched yet. The current free-tier models handle the volume comfortably. The bigger accuracy gains come from the GP scorer accumulating labelled examples, which costs nothing. Once the GP is active, the LLM only sees posts with a reasonable prior probability of being a lead — so model quality matters less at volume and more at the margin. That's the right time to upgrade.

06 — Limitations and roadmap

What it can't do yet

Honest about current gaps. These are engineering constraints, not fundamental problems.

🌐LinkedIn requires a residential machine
LinkedIn actively blocks cloud datacenter IPs at the network level. It is infrastructure-level blocking, not a policy that can be worked around. The Cloudflare Tunnel setup works well but it means the system is not fully autonomous since a local PC needs to be running. The right long-term fix is LinkedIn Sales Navigator API access at $80 per month, or using the Voyager internal API, both of which remove the Playwright browser dependency entirely.
🧠GP scorer is waiting for data
The self-learning component needs 10 labelled decisions before it switches on. Every accept or reject click in the dashboard review panel contributes toward that count. Once active it pre-filters posts before they reach the LLM, which reduces cost and handles the ambiguous edge cases the AI model currently gets wrong. This is the most impactful unactivated feature in the system right now.
📧Email enrichment only works for some leads
Hunter.io email lookup only works when the lead's company domain can be clearly identified from their post or profile. Most LinkedIn posts do not include a company website link. The outreach pipeline is built and tested. It just cannot complete automatically for the majority of leads yet. Integrating Apollo or Clay would improve email discovery coverage significantly.
🎯AI scoring still makes mistakes on edge cases
Technical founders describing scaling problems, posts that clearly describe a business gap without explicitly mentioning software development, these are genuinely hard to classify from text alone. The GP scorer once trained on real accept and reject data from this specific business context will handle these far better than the zero-shot LLM approach currently used for all posts.
