01 — The problem

Finding real leads is genuinely hard

Most lead generation tools search for keywords. Keywords attract the wrong people. This is the story of figuring out a better way.

What we tried first
Keyword search across platforms
Starting in n8n, we searched LinkedIn and other platforms with terms like "looking for developer", "need software built", "hire app team". Logical starting point.
  • Results were overwhelmingly agencies promoting services
  • Developers writing about their own projects ranked highly
  • Job listings mixed in constantly
  • Actual buyers rarely write the words you search for
  • AI had to score hundreds of irrelevant posts every run
The insight that changed everything
Feed-based discovery (from OpenOutreach)
Studying the open-source OpenOutreach project revealed a fundamentally different approach: scroll through the LinkedIn feed, not search results.
  • Feed posts come from actual founders and business owners
  • People describe real problems they are stuck on right now
  • Context is visible — who they are, what they run, what stage they're at
  • These posts never contain "hire developer" — they contain business problems
  • Lead quality on the very first test run improved dramatically
The core insight

Real clients don't write "I'm looking for a developer." They write "our inventory system is a spreadsheet and we have 12 staff now and it's breaking down" or "we need to digitise this entire workflow before Q3." Keyword search will never find these people. Feed browsing surfaces them naturally — the same way a human would spot an opportunity while scrolling. We built a system to do that at scale.

02 — The journey

From n8n to a production Python system

Each iteration revealed a new constraint. Here is what changed, why, and what the outcome was.

v1 · n8nStarted with n8n — worked, but hit a ceiling fast

n8n was the right starting point for quick automation without infrastructure. We connected LinkedIn search, basic scoring, and Supabase storage. But as soon as we needed custom scoring logic, ML models, or proper concurrency, n8n's node-based model became a constraint. Every edge case needed a workaround. There was no way to add a feedback loop or train anything from historical decisions.

DecisionRebuilt from scratch in Python with FastAPI. Full control over scoring logic, async execution, and the ability to add ML components properly.
v1–v2Keyword search kept returning agencies, not clients

Both in n8n and early Python, searching LinkedIn with "hire developer" or "need software built" returned results dominated by agencies self-promoting, developers posting about their own work, and job listings. After scoring hundreds of irrelevant posts, the pattern was clear: real buyers don't phrase their need as a search query. They describe a situation.

Turning pointFound OpenOutreach, an open-source project that scrolls the LinkedIn feed instead of searching. Adapted that approach using Playwright browser automation. First run returned noticeably different — and better — leads.
v2 · scoringScoring 60 leads was taking 5+ minutes and stalling on rate limits

The original scorer sent one post per API call, with a 2.1s sleep between calls to avoid rate limits on Groq's free tier. 60 leads meant 60 calls and 126 seconds of pure sleep time. Even with concurrency added, hitting Groq's 30 RPM limit caused retry loops that stalled the whole pipeline. Switching to NVIDIA still hit limits when running 6 concurrent calls.

FixBatch scoring: send 5 leads in one prompt, get back a JSON array of 5 scores. 60 leads → 12 API calls. Semaphore(3) for controlled concurrency. asyncio.gather() to run batches in parallel. Total scoring time: ~45 seconds. NVIDIA Nemotron-49B as primary, Groq Llama-70B as automatic fallback.
v2 · UXLinkedIn ran silently for 5 minutes — impossible to tell if it was working

The LinkedIn scraper runs as a subprocess (Playwright browser). The original implementation used subprocess.run(), which buffers all output and only delivers it after the process exits. A 3–5 minute browser run meant the dashboard log panel showed nothing for minutes. Users assumed the pipeline had crashed.

FixReplaced subprocess.run() with subprocess.Popen() + line-by-line reading. Each output line is forwarded to the async event loop immediately via loop.call_soon_threadsafe(). Now every scroll action and collected post appears in the dashboard log in real time as it happens.
v2 · deployLinkedIn on cloud simply does not work — blocked by design

After deploying to Railway, LinkedIn failed completely. Two separate issues: session files were gitignored so Railway had nothing to authenticate with, and LinkedIn's anti-bot system actively blocks datacenter IP ranges from AWS, Railway, and Render. Even with session files present, the IP would get flagged immediately.

Architecture decisionLinkedIn must run on a residential IP (home/office connection). Current setup: Cloudflare Tunnel runs on a local PC and exposes a permanent public HTTPS URL. LinkedIn sees a normal residential IP. Reddit and Twitter run on Railway without issue since they don't have this restriction.
ongoingEdge cases in AI scoring — technical founders, indirect signals

Feed-based scraping dramatically improved signal quality but the scoring model still has edge cases. A technical founder describing a scaling challenge can look like a developer to the AI. Posts that describe a business problem without explicitly asking for software development are genuinely ambiguous. These require human judgement.

Long-term fixThe GP (Gaussian Process) scorer learns from every accept/reject click in the dashboard. Once 10 labelled examples are accumulated, it activates and starts pre-filtering posts before they reach the LLM — reducing cost and improving accuracy on exactly the edge cases the LLM struggles with. This gets smarter the more it's used.

Batch scoring — the key optimisation
Why 5 leads per call design decision
Sending each lead individually meant 60+ API calls per run, constant rate limit hits, and sequential waits. Batching 5 leads into one prompt with a structured JSON response format cuts calls to 12 and fits comfortably within any free-tier limit. The model returns a JSON array — one score and reasoning per lead — which gets parsed and matched back to the original posts.
60 leads → 12 calls → fits 30 RPM free tier with room to spare
Concurrency model
asyncio.Semaphore(3) limits simultaneous API calls. asyncio.gather() runs all 12 batches concurrently within that limit. A single shared httpx.AsyncClient is reused across all calls, eliminating per-call TCP connection overhead. The same gather pattern applies to Supabase writes — all leads are saved in parallel, not sequentially.
Semaphore(3) × 12 batches = wall time of ~4 batches sequentially
SSE vs WebSocket design decision
Log streaming uses Server-Sent Events over a persistent HTTP connection. SSE is unidirectional (server → browser only), which is exactly what's needed here. No handshake overhead, no socket management, works through proxies, and the browser's EventSource API handles reconnection automatically. WebSockets would add complexity for no benefit.
Single HTTP connection, EventSource handles reconnect
Self-learning GP scorer
Every accept/reject click saves a (post_text, label) pair. The GP scorer converts post text to 384-dimensional embeddings using FastEmbed, then trains a GaussianProcessRegressor with an RBF kernel. After 10 labels, it activates and assigns probability scores to new posts before they reach the LLM. Posts below 0.2 probability are skipped entirely, saving API calls on low-signal content.
FastEmbed 384-dim → GP(RBF kernel) → P(lead) → skip or score

# 5 leads per call. Model returns a JSON array.
def _build_batch_prompt(leads: list) -> str:
    SYSTEM = """Score each post 1-10 for software development intent.
Immediately score 1 if the author is a developer, agency, writing a success
story, posting a job, or producing promotional content.
Score 9-10: explicitly requests software to be built.
Score 7-8:  business problem that clearly requires a software solution.
Return JSON array only: [{"id":0,"score":8,"reason":"..."},...]"""

    posts_block = "\n\n".join(
        f"POST {i}: {lead['text'][:800]}"
        for i, lead in enumerate(leads)
    )
    return f"{SYSTEM}\n\n{posts_block}"

# Primary: NVIDIA. Fallback: Groq. Transparent to caller.
async def _score_batch(leads: list) -> list:
    async with _sem:  # Semaphore(3)
        result = await _call_nvidia(leads)
        if result is None:   # 429 or error
            result = await _call_groq(leads)
        return result

# All batches run concurrently within the semaphore limit
batches = [leads[i:i+5] for i in range(0, len(leads), 5)]
results = await asyncio.gather(*[_score_batch(b) for b in batches])

# Playwright runs in a subprocess. Popen() gives us lines as they arrive.
# loop.call_soon_threadsafe() bridges the blocking thread to the async loop.
def _stream_subprocess(process, loop, log_fn):
    for line in process.stdout:          # blocks in thread — that's fine
        loop.call_soon_threadsafe(log_fn, line.strip())

async def fetch_linkedin_leads(mode="hot", log=print):
    loop = asyncio.get_event_loop()
    process = subprocess.Popen(
        [sys.executable, "-u", scraper_path],  # -u = unbuffered stdout
        stdout=PIPE, stderr=STDOUT, text=True, bufsize=1
    )
    # Run blocking readline loop in thread pool — doesn't block event loop
    await loop.run_in_executor(None, _stream_subprocess, process, loop, log)

04 — AI models

Current setup and what a better model unlocks

The system is model-agnostic by design. Swapping providers is a one-line change. Here is where the current model does well and where it falls short.

Currently in production
NVIDIA Nemotron-49B
nvidia/llama-3.3-nemotron-super-49b-v1.5 · free via NVIDIA NIM
  • No billing required — starts immediately on a free API key
  • Handles the 5-lead batch format reliably, consistent JSON output
  • Good baseline accuracy on clear-cut cases (explicit "build me X" posts)
  • Falls back to Groq Llama-3.3-70B automatically on 429
  • Struggles with indirect signals and ambiguous founder context
  • Context window caps how long each post can be before trimming
Where Claude / Gemini would improve this
Claude Sonnet or Gemini Pro
Anthropic / Google · paid or generous free tiers
  • 200k token context — can pass the full post, author bio, and company info without trimming
  • Much better at indirect signals: "we're manually running this in spreadsheets for 200 clients" → lead, not just noise
  • Claude follows structured scoring instructions with notably fewer format errors
  • Gemini Pro: 1 million tokens/day free — easily enough for current volume without billing
  • Both support system-level calibration prompts that can be tuned from real false-positive data
  • Better false-reject rate on technical founders who also happen to write about code
Why not switched yet. The current free-tier models handle the volume comfortably. The bigger accuracy gains come from the GP scorer accumulating labelled examples, which costs nothing. Once the GP is active, the LLM only sees posts with a reasonable prior probability of being a lead — so model quality matters less at volume and more at the margin. That's the right time to upgrade.

06 — Limitations and roadmap

What it can't do yet

Honest about current gaps. These are engineering constraints, not fundamental problems.

🌐LinkedIn requires a residential machine
LinkedIn actively blocks cloud datacenter IPs at the network level. Not a policy issue — it's infrastructure-level blocking. The Cloudflare Tunnel workaround works well but means the system isn't fully autonomous. A local PC needs to be running. The right long-term fix is LinkedIn Sales Navigator API access ($80/month) or the Voyager internal API — both remove the Playwright dependency entirely.
🧠GP scorer is waiting for data
The self-learning component needs 10 labelled decisions to activate. Every review in the dashboard contributes. Once active, it pre-filters posts before they hit the LLM — reducing cost and handling the ambiguous edge cases the AI model currently struggles with. This is the most impactful unactivated feature in the system right now.
📧Email enrichment is partial
Hunter.io email lookup works for roughly 30% of leads — only when the lead's company domain is clearly identifiable. Most LinkedIn posts don't include a company website. The outreach step is built; it just can't complete automatically for the majority of leads yet. Apollo or Clay integrations would improve coverage significantly.
🎯Still some false positives at the edges
Technical founders who write about scaling problems, posts that describe a clear business gap without explicitly asking for software — these are genuinely hard to classify without broader context. The GP scorer, once trained on real accept/reject data from this specific business context, will handle these far better than a zero-shot LLM.
